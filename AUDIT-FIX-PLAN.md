# G-code viewer — audit findings & fix plan

Audit of the full library (TypeScript renderer + Rust/WASM parser) at commit `20f0688`
(fork is currently in sync with upstream `Duet3D/gcodeviewer` master).

**Status: all items below (1–28, 30) have been implemented in the working tree.** Items 29, 31,
32 were intentionally left for a follow-up session that can drive a real browser (see the note
at the end of this file). `npm run check` and `npm run build` both pass clean on this machine,
which has no Rust/wasm-pack toolchain — that itself exercises the item 1 fix. The Rust-side
changes (items 1, 6, 12, 13, 15) could not be compiler-verified here (no `cargo`/`wasm-pack`
available); run `npm run build:wasm` on a machine with the Rust toolchain and `cargo test` inside
`WASM_FileProcessor/` before shipping, and treat those as needing a closer look in review.

**How to use this document:** each numbered item is an independent, implementable fix with
file references, the problem, the proposed fix, and how to verify. Work top-down: P0 items
are crashes/build breaks, P1 is the picking/render-pipeline area (the known "viewer vomits
on itself" bug), P2 is correctness, P3 is hygiene/leaks. Do **not** batch unrelated items
into one commit — one item (or one tightly related group) per commit.

## P0 — Crashes and build breaks

### 1. Repo does not typecheck or build without the Rust toolchain
- **Files:** [src/wasmprocessor.ts:1](src/wasmprocessor.ts#L1), [src/processor.ts:88](src/processor.ts#L88), [vite.config.mjs](vite.config.mjs)
- **Problem:** both files import `../WASM_FileProcessor/pkg/gcode_file_processor`, which is
  generated by `wasm-pack` and is not committed. `npm run check` fails with TS2307 on a clean
  clone, and `npm run build` aborts unless `wasm-pack` + the `wasm32-unknown-unknown` target
  are installed (the vite plugin tries to build it).
- **Fix:** pick one (a) is preferred:
  (a) commit a checked-in `WASM_FileProcessor/pkg/` (built artifact) and have the vite plugin
  only rebuild when the toolchain exists, warning otherwise; or
  (b) add a hand-written `.d.ts` ambient module declaration for the pkg and make both imports
  lazy `await import(...)` inside `WasmProcessor.initialize()` so the module is only touched
  when WASM is enabled.
  Either way, `getWasmVersion()` in processor.ts should go through `WasmProcessor` instead of
  doing its own second import of the pkg.
- **Verify:** `npm run check` and `npm run build` succeed on a machine without cargo/wasm-pack.

### 2. `Processor.modelMaterial` is never initialized — any API call before the first file load crashes the worker
- **File:** [src/processor.ts:29](src/processor.ts#L29)
- **Problem:** `modelMaterial: LineShaderMaterial[]` has no initializer; it only becomes an
  array inside `cleanup()` (called from `loadFile`). Calling `setTools`, or the worker messages
  `rendermode`, `setalphamode`, `setprogressmode`, `setmeshmode` (which calls
  `refreshMaterialState` via `setMeshMode`) before a file is loaded throws
  `TypeError: Cannot read properties of undefined (reading 'forEach')` inside the worker.
  DWC sends tool/config state on connect, before a file is opened, so this fires in practice.
- **Fix:** initialize `modelMaterial: LineShaderMaterial[] = []` at the declaration.
- **Verify:** create viewer, call `setTools()` / `setRenderMode()` before `loadFile()` — no error.

### 3. Worker message `updatecolortest` calls a method that does not exist
- **Files:** [src/viewer.worker.ts:61](src/viewer.worker.ts#L61), [src/viewer-proxy.ts:199](src/viewer-proxy.ts#L199)
- **Problem:** `processor.updateColorTest()` does not exist on `Processor`; the async message
  handler throws (unhandled rejection in the worker) whenever a consumer calls
  `ViewerProxy.updateColorTest()`.
- **Fix:** remove the worker case and the proxy method (it is a leftover of the v3 API), or
  implement it if DWC still calls it (search DWC plugin code before deciding; removal preferred).
- **Verify:** grep for remaining references; worker no longer has an unreachable case.

### 4. `T` line handler crashes on Klipper-style macros and unknown tool numbers
- **File:** [src/GCodeCommands/t.ts](src/GCodeCommands/t.ts)
- **Problem (two bugs):**
  1. `ProcessLine` routes **any** line starting with `T` here
     ([processline.ts:170-172](src/GCodeCommands/processline.ts#L170-L172)). For Klipper macros
     (`TIMELAPSE_TAKE_FRAME`, `TURN_OFF_HEATERS`, …) `line.match(toolRegex)` returns `null` and
     `[0]` throws. `loadFileStreamed` has **no** per-line try/catch, so the whole file load dies.
  2. `props.tools[toolIdx]` with `toolIdx >= tools.length` (e.g. `T5` with the default 5-tool
     table) sets `currentTool = undefined`; the next move line then throws on
     `props.currentTool.toolNumber`.
- **Fix:** if the regex does not match, return `new Comment(props, line)`. Clamp/modulo the tool
  index or fall back to tool 0 when out of range (matching the Rust side's "create tool if
  missing" behavior is also acceptable).
- **Verify:** load a G-code file containing `TIMELAPSE_TAKE_FRAME` and `T7` — file loads.

### 5. Fast-path parser misparses `G10 <params>` as a G1 move (position corruption)
- **File:** [src/GCodeCommands/processline.ts:175-189](src/GCodeCommands/processline.ts#L175-L189)
- **Problem:** the fast-path guard is
  `(workingLine[1]==='0'||workingLine[1]==='1') && (workingLine[2]===' '||workingLine[2]==='0')`.
  `"G10 P1 X5 …"` satisfies it (`[1]==='1'`, `[2]==='0'`), and `parseG0G1Fast` then reads the
  `1` after `G` as "is G1" and applies the X/Y/Z tokens as a **move**, mutating
  `props.currentPosition`. G10 with parameters (tool/workplace offsets, CNC `G10 L2 …`) corrupts
  every subsequent absolute position. (Bare `G10` retraction lines are unaffected.) The same
  guard also *misses* `G01 …` (falls back to the slow path — a perf quirk, not a bug).
- **Fix:** make the guard exact: command token must be `G0`, `G1`, `G00` or `G01` followed by
  whitespace or end-of-token (mirror `is_g0_g1_command` in
  [WASM_FileProcessor/src/GCodeCommands/G0G1.rs:187-213](WASM_FileProcessor/src/GCodeCommands/G0G1.rs#L187-L213)).
  Additionally `parseG0G1Fast` must bail (`return null`) when `props.zBelt` is set — the fast
  path has no belt-kinematics handling while `g0g1.ts` does (see item 19).
- **Verify:** unit-style check: process `G10 P1 X0 Y0`, assert `currentPosition` unchanged and
  the returned object is not a move; process `G01 X5` and assert it is.

### 6. Empty/tiny files crash both pipelines
- **Files:** [src/processor.ts:226-229](src/processor.ts#L226-L229), [src/processor.ts:820-824](src/processor.ts#L820-L824), [src/processor.ts:826-856](src/processor.ts#L826-L856), [WASM_FileProcessor/src/processor.rs:53](WASM_FileProcessor/src/processor.rs#L53)
- **Problem:**
  1. TS: `this.gCodeLines[this.gCodeLines.length - 1].filePosition` throws when the file is
     empty (`loadFile`, `getFileSize`); `getGCodeInRange` indexes `gCodeLines[-1]` when empty.
  2. Rust: `chunk_size = 10000.min(estimated_lines / 10)` is **0** for files under ~400 bytes,
     and `lines_processed % chunk_size` then panics (remainder by zero) on the first line. With
     `panic = "abort"` the WASM instance is poisoned; every later WASM load fails until reload
     (it does fall back to TypeScript, but noisily and permanently).
- **Fix:** guard the TS accesses on `gCodeLines.length === 0` (post `fileloaded` with 0/0 and
  return early); in Rust use `chunk_size.max(1)` (and same for any other computed divisor).
- **Verify:** load `""`, `"G28\n"`, and a 200-byte file with WASM enabled — no crash, WASM stays
  usable for the next file.

### 7. Null slicer and reload-before-load guards
- **Files:** [src/GCodeParsers/slicerfactory.ts:38](src/GCodeParsers/slicerfactory.ts#L38), [src/processor.ts:1051-1054](src/processor.ts#L1051-L1054)
- **Problem:** `slicerFactory(null)` returns `null`, and `Move`'s constructor dereferences
  `props.slicer.getFeatureColor()`. `setPerimeterOnly()` calls `loadFile(this.originalFile)`
  which is `undefined` before any load — same crash path.
- **Fix:** `slicerFactory` returns `new GenericBase()` for null/empty input; `setPerimeterOnly`
  no-ops when `originalFile` is unset.
- **Verify:** toggle perimeter-only before loading a file — no error.

## P1 — Picking / render pipeline (the reported "disable picking → viewer vomits on itself" bug)

Context: there is **no** picking enable/disable API in the current code — picking is
constructed unconditionally and, more importantly, it is *structurally load-bearing*. The
reported corruption is almost certainly not one exotic GPU bug but the combination of the
following: any plausible way of "disabling picking" (not constructing `GPUPicker`, disposing
the RTT, or removing it from `scene.customRenderTargets`) either throws mid-`loadFile` and
leaves the scene half-built, or leaves per-frame observables mutating state they no longer
own. Items 8–11 together make picking a first-class, cleanly detachable feature; item 9 also
fixes real GL-state violations that exist even with picking *enabled*.

### 8. Make picking optional and null-safe (root-cause fix for the reported bug)
- **Files:** [src/processor.ts](src/processor.ts) (lines 125, 207, 229, 597, 613, 664, 685, 861, 965, 1174), [src/viewer.ts:116](src/viewer.ts#L116), [src/viewer.ts:145-150](src/viewer.ts#L145-L150), [src/gpupicker.ts](src/gpupicker.ts)
- **Problem:** `Processor` and `Viewer` dereference `this.gpuPicker` unconditionally in ~9
  places: `cleanup()` (first statement of `loadFile`!), both mesh-build loops, `loadFile`'s
  post-processing, `updateFilePosition`, `animateToNextPosition`, and `setSizes`. If the picker
  is absent or disposed, `loadFile` throws **after** meshes/materials were partially created and
  **before** `fileloaded`/material-uniform updates are sent — stale chunks of the previous model
  plus new half-initialized meshes with default uniforms render on screen: the "vomit". The
  same class of breakage occurs on resize (`setSizes` → `updateRenderTargetSize` on a disposed
  RTT throws after `engine.resize()`).
- **Fix:**
  1. Add `setPickingEnabled(enabled: boolean)` on `Viewer` + worker message + proxy method.
  2. Disable path: remove the RTT from `scene.customRenderTargets`, clear
     `renderTarget.renderList`/`renderTargetMeshs`, do **not** dispose the shader material (so
     re-enable is cheap), and stop the observables from doing work (early-return flag).
  3. Enable path: re-add the RTT, re-register the current meshes via `addToRenderList`.
  4. Make every `gpuPicker` call site null/disabled-safe (`this.gpuPicker?.…` or an internal
     no-op picker), so the processor pipeline is fully independent of picking.
  5. Wrap the per-line/mesh loop bodies of `loadFile` in a try/finally that always posts
     `fileloaded`/`progress:1` so the UI can never be left hanging by a mid-load exception.
- **Verify:** load a file, call `setPickingEnabled(false)`, load another file, resize, change
  mesh mode, scrub position, re-enable picking and hover — rendering stays correct at every step.

### 9. GPUPicker leaks a framebuffer per frame, mutates GL state behind Babylon's back, and stalls the pipeline every frame
- **File:** [src/gpupicker.ts:53-116](src/gpupicker.ts#L53-L116)
- **Problem (several, all in the same rework):**
  1. `readTexturePixels` calls `gl.createFramebuffer()` **every frame** and never deletes it —
     unbounded GPU object leak (~60 FBOs/second while the tab is visible).
  2. It never restores the previously bound framebuffer (raw `gl.bindFramebuffer` behind
     Babylon's state cache). It currently *happens* to recover because Babylon rebinds
     afterwards, but it is fragile across Babylon versions.
  3. It reads `this.renderTarget._texture._hardwareTexture.underlyingResource` — triple
     private-API access; if `_texture` is transiently null (during `resize()`), the observable
     throws **inside `scene.render()`**, and an exception there kills Babylon's render loop
     entirely (frozen canvas — another credible "vomit" path).
  4. A synchronous `gl.readPixels` runs **every frame**, even when the pointer hasn't moved —
     a full GPU pipeline stall per frame.
  5. The RTT is constructed with `generateMipMaps = true` and `refreshRate = 1`, so mipmaps are
     regenerated every frame for a texture that is only ever point-sampled.
  6. `console.log(this.scene.customRenderTargets)` debug noise in the constructor.
- **Fix (rework `GPUPicker`):**
  - Construct the RTT with `generateMipMaps: false`.
  - Only render/read on demand: track pointer moves (the viewer already receives them); set
    `renderTarget.refreshRate = RenderTargetTexture.REFRESHRATE_RENDER_ONCE` and call
    `renderTarget.render()`/mark dirty on a throttled pointer-move, or keep refreshRate 1 but
    skip the `readPixels` when the pointer hasn't moved since the last read.
  - Create **one** framebuffer in the constructor, reuse it, delete it in a new `dispose()`;
    save/restore the previous binding (`gl.getParameter(gl.FRAMEBUFFER_BINDING)`), and guard
    `renderTarget._texture` being null. Prefer Babylon's supported
    `engine._readTexturePixels`/`readPixels` helpers over raw GL if workable.
  - Wrap the observable bodies in try/catch so a picking failure can never take down
    `scene.render()`.
  - **Alternative worth evaluating:** Babylon ≥7.31 ships a built-in `GPUPicker` with thin
    instance support; replacing this custom class removes the whole category. Evaluate before
    hand-fixing, but do not block the fixes above on it.
- **Verify:** with picking enabled, hover works; `gl.createFramebuffer` is called once (spy or
  spector.js capture); no per-frame readback when the mouse is idle (frame time drops).

### 10. Per-frame `setEnabled` churn from the pick pass
- **File:** [src/gpupicker.ts:52-82](src/gpupicker.ts#L52-L82)
- **Problem:** every frame the picker force-enables all registered box meshes before its pass
  and re-disables them after (when mesh mode is cylinder/line). `setEnabled` dirties Babylon's
  active-mesh caches scene-wide, twice per frame, and the "was it enabled" snapshot comes from
  `renderTargetMeshs[0]` only.
- **Fix:** stop toggling `setEnabled`. Since the pick pass has its own render list and material,
  the meshes don't need to be scene-enabled to render into the RTT **if** picking always
  registers the currently-active mesh set instead: register the active mesh (box/cyl/line per
  `setMeshMode`) in the pick render list rather than always the box. Simplest correct version:
  in `Processor.setMeshMode`, re-populate the picker's render list with the meshes that are now
  enabled. Then delete the enable/disable observables entirely.
- **Verify:** picking works in all three mesh modes; no `setEnabled` calls per frame (breakpoint).

### 11. Picking tap handler uses a magic `> 10` threshold and swallows errors
- **File:** [src/viewer.ts:249-256](src/viewer.ts#L249-L256), [src/processor.ts:207-221](src/processor.ts#L207-L221)
- **Problem:** `focusedColorId > 10` arbitrarily makes the first 10 file lines unclickable and
  papers over the id-0 "background" sentinel; the empty `catch {}` hides real errors; a stale
  `focusedColorId` survives file reloads (`cleanup()` doesn't reset it, nor `filePosition`).
- **Fix:** use `-1` as the "nothing hovered" sentinel (background reads as 0 → id `-1` after the
  `-1` offset); check `id >= 0 && id < gCodeLines.length`; reset `focusedColorId`/`filePosition`
  in `cleanup()`; log rather than swallow exceptions.
- **Verify:** clicking the very first extrusion of a file jumps to it; clicking empty space does
  nothing; after loading a second file, the first click doesn't jump to a stale position.

## P2 — Correctness (parsing and WASM/TS parity)

### 12. CRLF files: WASM file positions drift one byte per line
- **Files:** [WASM_FileProcessor/src/processor.rs:69-183](WASM_FileProcessor/src/processor.rs#L69-L183), [src/processor.ts:1275-1293](src/processor.ts#L1275-L1293)
- **Problem:** Rust `str::lines()` strips `\r\n` but the position math adds `line.len() + 1`,
  so on CRLF files (most Windows-slicer output) every line drifts `filePosition` by +1
  cumulatively. TS `streamLines` keeps the `\r` so its math is correct. Result: the WASM render
  buffers' `filePosition` attributes disagree with the TS `gCodeLines` used for
  picking/scrubbing — hover shows the wrong line, position scrub cuts the model at wrong places.
- **Fix:** in Rust, split on `\n` manually (keeping `\r` in the slice like TS does), or compute
  the true consumed byte count per line. Add a CRLF fixture test in `tests.rs` asserting
  positions match byte offsets.
- **Verify:** load a CRLF-converted file with WASM enabled; hover a line near the end of file
  and confirm the reported `lineNumber` matches the actual file line.

### 13. Absolute-positioning default disagrees: TS `false`, Rust `true`
- **Files:** [src/processorproperties.ts:46](src/processorproperties.ts#L46), [WASM_FileProcessor/src/processor_properties.rs:194](WASM_FileProcessor/src/processor_properties.rs#L194)
- **Problem:** G-code default is absolute (G90); TS defaults to relative. Files that never issue
  G90 (some post-processors strip it) parse as garbage in the TS path, and — worse — the hybrid
  path parses the *same file* with different defaults for buffers (Rust) vs `gCodeLines` (TS).
- **Fix:** default TS `absolute = true`. Keep `extruderAbsolute` semantics unchanged.
- **Verify:** file with no G90 renders identically in WASM and TS paths.

### 14. WASM and TS paths color extrusions differently
- **Files:** [src/GCodeCommands/g0g1.ts:26-30](src/GCodeCommands/g0g1.ts#L26-L30) (tool color), [WASM_FileProcessor/src/GCodeCommands/G0G1.rs:130-134](WASM_FileProcessor/src/GCodeCommands/G0G1.rs#L130-L134) (feature color)
- **Problem:** for G1 moves TS writes `currentTool.color` into `move.color` (the `baseColor`
  attribute used by renderMode 0), while Rust writes `current_feature_color`. The same file
  loads with different colors depending on which pipeline produced the buffers.
- **Fix:** decide the intended semantics for renderMode 0 (feature color is what slicer-aware
  viewers show; tool color is already available via renderMode 1) and make both sides match —
  recommendation: **feature color** in `baseColor` on both sides, drop the tool-color overwrite
  in `g0g1.ts`/`parseG0G1Fast`.
- **Verify:** load the same PrusaSlicer file via WASM and via TS fallback — identical colors.

### 15. WASM arc tessellation keys collide with later lines
- **File:** [WASM_FileProcessor/src/processor.rs:146-167](WASM_FileProcessor/src/processor.rs#L146-L167)
- **Problem:** arc segments are inserted into `position_tracker` with key
  `file_position + seg_index`. Once `seg_index` exceeds the arc line's byte length, keys collide
  with the file positions of *subsequent lines*, silently overwriting their entries (HashMap
  insert). Long arcs corrupt positions/animation for following moves.
- **Fix:** the tracker key must remain the line's `file_position`; store arc segments in a
  separate ordered vector (or make the tracker value a small Vec of segments). Update
  `generate_render_buffers` to iterate segments explicitly instead of relying on synthetic keys,
  and set `file_position_data` for every arc segment to the arc line's real position.
- **Verify:** file with G2/G3 arcs (>60-segment arc) — hovering lines *after* the arc reports
  correct line numbers; segment count matches the TS path.

### 16. WASM-built meshes miss the thin-instance bounding/static setup (model can vanish)
- **File:** [src/processor.ts:1028-1049](src/processor.ts#L1028-L1049) vs the TS path at [src/processor.ts:791-805](src/processor.ts#L791-L805)
- **Problem:** `applyWasmBuffersToMesh` passes `staticBuffer = false` to every
  `thinInstanceSetBuffer` (per-frame re-upload cost) and never calls
  `doNotSyncBoundingInfo = true` + `thinInstanceRefreshBoundingInfo(false)` nor
  `isPickable = false`. The mesh's bounding box stays the unit box at the origin, so frustum
  culling hides the entire model at many camera angles.
- **Fix:** mirror the TS `copyBuffers` exactly: static buffers (`true`), refresh thin-instance
  bounding info, `doNotSyncBoundingInfo = true`, `isPickable = false`. Drop the redundant
  `thinInstanceCount` assignment (setting the matrix buffer already sets it).
- **Verify:** WASM-loaded model stays visible when zoomed in so the origin is off-screen.

### 17. Mesh generation skips the last G-code line
- **Files:** [src/processor.ts:568](src/processor.ts#L568), [src/processor.ts:634](src/processor.ts#L634)
- **Problem:** both loops iterate `idx < this.gCodeLines.length - 1`, so the final line of the
  file is never rendered (visible as a missing final segment; also skews perimeterOnly).
- **Fix:** iterate `idx < this.gCodeLines.length`.
- **Verify:** file whose last line is an extruding move renders that segment.

### 18. Arc parsing defects (`K` clobbers `J`; feed rate ignored; sloppy `E` detection)
- **Files:** [src/util.ts:65-69](src/util.ts#L65-L69), [src/GCodeCommands/g2g3.ts:13-17](src/GCodeCommands/g2g3.ts#L13-L17)
- **Problem:**
  1. `doArc`'s `case 'K'` writes into `j`, overwriting a preceding `J` token — YZ-plane (G19)
     arcs with `J`+`K` parse wrong. (XZ works only by accident of the later i/j swap.)
  2. `g2g3` never reads `F`, so arcs don't update the feed rate (feed-rate render mode and
     animation timing wrong on arc-heavy files).
  3. `move.extruding = line.indexOf('E') > 0` matches an `E` anywhere — including comments
     (`G2 X1 Y1 I1 ; Edge`) and negative retractions (`E-2`).
- **Fix:** track `i`/`j`/`k` separately in `doArc` and map them per `arcPlane`; parse `F` in
  `g2g3` like `g0g1` does; detect extrusion from a real `E<number>` token with value > 0 in the
  pre-comment part of the line.
- **Verify:** unit checks on `doArc` for all three planes; a G2 with `; Edge` comment is a
  travel unless it has positive E.

### 19. `parseG0G1Fast` ignores belt-printer (`zBelt`) kinematics
- **File:** [src/GCodeCommands/processline.ts:74-110](src/GCodeCommands/processline.ts#L74-L110) vs [src/GCodeCommands/g0g1.ts:33-63](src/GCodeCommands/g0g1.ts#L33-L63)
- **Problem:** the fast path applies plain X/Y/Z mapping even when `props.zBelt` is set, while
  the slow path applies the gantry-angle transform — belt files render differently depending on
  which path a line happens to take.
- **Fix:** bail out of the fast path (`return null`) when `props.zBelt` is true (covered
  together with item 5's guard rewrite).
- **Verify:** with `zBelt` forced on, both `G1 X10 Y10` and a complex G1 line produce identical
  positions.

### 20. `perimeterOnly` replaces comments/commands and drops travels
- **Files:** [src/processor.ts:635-639](src/processor.ts#L635-L639) (and same in `testRenderScene`)
- **Problem:** `if (this.perimeterOnly && !gCodeline.isPerimeter)` is evaluated for *every*
  line; comments/commands have no `isPerimeter` so they're needlessly rewrapped as `Move_Thin`,
  and travels are excluded from rendering entirely in perimeter mode. Feature flags from
  PrusaSlicer are also questionable (see item 22), making perimeter mode mostly show infill.
- **Fix:** apply the perimeter filter only to `lineType === 'L' | 'A' | 'T'`; keep travels
  subject to the same rule intentionally (decide + comment); leave non-moves untouched.
- **Verify:** perimeter-only toggle: comments remain `Comment` instances; model shows outer
  walls.

### 21. `CurrentFeedRate` min/max updates lag one assignment behind
- **File:** [src/processorproperties.ts:77-86](src/processorproperties.ts#L77-L86)
- **Problem:** the setter compares the **old** `currentFeedRate` before storing the new value,
  so the last-set feed rate is never folded into min/max, and the first real value compares
  against the initial `1`. Feed-rate render mode ranges are subtly wrong.
- **Fix:** compare and fold the **incoming** value: update `maxFeedRate`/`minFeedRate` from
  `value` (ignoring 0), then store.
- **Verify:** file with F1200/F3000 only → min 1200, max 3000 (not 1 / 1200).

### 22. PrusaSlicer parser: inverted-looking perimeter flags and dead/broken `processHeader`
- **File:** [src/GCodeParsers/prusaslicer.ts](src/GCodeParsers/prusaslicer.ts)
- **Problem:**
  1. `PERIMETER` has `perimeter: false` while `TOP SOLID INFILL` and `WIPE TOWER` have
     `perimeter: true` — the flag drives perimeter-only mode and the line-mode shading, and this
     mapping looks inverted/arbitrary. Same pattern in the other slicer parsers — review all.
  2. `processHeader` guard `if (props.tools.length < toolIdx)` is inverted (should be
     `toolIdx < props.tools.length`), and the method is never called anywhere.
- **Fix:** set `perimeter: true` for PERIMETER/EXTERNAL PERIMETER/OVERHANG PERIMETER and false
  for infill/support/wipe; either wire `processHeader` into the load path (it parses nozzle
  diameters) with the comparison fixed, or delete it.
- **Verify:** perimeter-only on a PrusaSlicer file shows walls, not infill.

### 23. Tool colors set via `setTools` are silently lost on every file load
- **Files:** [src/processor.ts:150](src/processor.ts#L150), [src/processorproperties.ts:100-104](src/processorproperties.ts#L100-L104), [src/tools.ts:13-19](src/tools.ts#L13-L19)
- **Problem:** `loadFile` recreates `ProcessorProperties`, whose constructor assigns the shared
  module-level `tools` array — DWC-provided tool colors/diameters from a prior `setTools` call
  are discarded; also the default `tools` array is a shared mutable singleton across instances.
- **Fix:** keep the externally-provided tool table on the `Processor` (e.g. `userTools`) and
  re-apply it after recreating properties in `loadFile`; construct fresh default `Tool` objects
  per `ProcessorProperties` instead of sharing the singleton.
- **Verify:** `setTools([...])` → `loadFile(...)` → tool-color render mode uses the set colors.

### 24. Bed placement and material-disposal bugs
- **File:** [src/Renderables/bed.ts](src/Renderables/bed.ts)
- **Problem:**
  1. Box mode (non-delta): `position.x = bedCenter.x - buildVolume.x.min` (and same for y/z)
     misplaces the volume box for printers whose axes don't start at 0 (center-origin CoreXY:
     min −150 → box shifted +150).
  2. Delta box mode: `position.z = bedCenter.x` should be `bedCenter.y`.
  3. `dispose()` calls `bedMesh.dispose(false, true)` which disposes the **shared**
     `planeMaterial`/`boxMaterial`, but `buildBed()` then reuses those disposed instances
     (`setRenderMode`, `commitBedSize` — called by `setBuildVolume` —, `setDelta` all hit this).
- **Fix:** box position = `bedCenter` (x/z from center, y = center of z-range); fix the delta
  z-coordinate; dispose meshes with `dispose(false, false)` and manage the two materials'
  lifetimes explicitly (dispose them only in a class-level `dispose`).
- **Verify:** `setBuildVolume({x:{min:-150,max:150}, …})` + box render mode centers the box on
  the bed; toggling bed render mode repeatedly keeps the grid visible.

## P3 — Lifecycle, leaks, API hygiene

### 25. Unload/teardown leaks on the main thread and in renderables
- **Files:** [src/viewer-proxy.ts:94-100](src/viewer-proxy.ts#L94-L100), [src/viewer-proxy.ts:130-152](src/viewer-proxy.ts#L130-L152), [src/Renderables/buildobjects.ts:84-93](src/Renderables/buildobjects.ts#L84-L93), [src/Renderables/nozzle.ts:112-121](src/Renderables/nozzle.ts#L112-L121)
- **Problem:** the proxy assigns `window.onresize = …` (clobbers the host app's handler and any
  second viewer's; never removed); every DOM listener registered on window/document/canvas for
  the worker is anonymous and never removed on `unload` (and they call `preventDefault` on
  document-level events forever after the worker is terminated); `BuildObjects.rebuildMaterials`
  creates 4 materials + a DynamicTexture per `loadObjectBoundaries` call without disposing the
  previous ones (DWC calls this on job updates); `Nozzle.setDiameter` leaks the hot-end material.
- **Fix:** use `window.addEventListener('resize', …)` and keep references to every listener the
  proxy registers (target, event, fn, opt) so `unload()` can remove them all; dispose old
  materials/texture in `rebuildMaterials`; dispose the hot-end material in `Nozzle.dispose`.
  Also stop `Processor` timers (`playbackTimeout`) in `Viewer.unload()`.
- **Verify:** create + unload the viewer twice on one page; no duplicated listeners (DevTools
  `getEventListeners`), keyboard/scroll work again after unload.

### 26. API-surface inconsistencies
- **Files:** [src/viewer-proxy.ts:183-197](src/viewer-proxy.ts#L183-L197), [src/viewer.worker.ts](src/viewer.worker.ts), [src/viewer-direct.ts](src/viewer-direct.ts), [src/index.ts](src/index.ts)
- **Problem:** proxy sends `cancel`/`reset` messages that the worker never handles; the direct
  (non-worker) viewer is missing most of the proxy API (tools, bed, build volume, nozzle,
  viewbox, WASM enablement, background color, clip planes…) so swapping the export breaks
  consumers; `index.ts` exports only `Viewer_Proxy`.
- **Fix:** remove `cancel`/`reset` (or implement); either bring `ViewerDirect` to parity by
  generating both from a shared method list, or mark it explicitly unsupported; export both
  classes deliberately.
- **Verify:** compile-time: a shared interface type implemented by both proxy and direct.

### 27. `cloneEvent` relies on `constructor.name` (breaks under minification)
- **File:** [src/viewer-proxy.ts:359-366](src/viewer-proxy.ts#L359-L366)
- **Fix:** `event instanceof KeyboardEvent` instead of comparing `constructor.name` (class names
  are commonly mangled in consuming apps' production builds → keyboard events get cloned with
  the mouse field list and camera keyboard control breaks only in production).
- **Verify:** keyboard camera controls work in a minified consumer build.

### 28. Nozzle position lookup is O(n) per scrub event
- **File:** [src/processor.ts:880-931](src/processor.ts#L880-L931)
- **Problem:** `updateNozzlePositionInstant/Animated` linearly scan the whole `positionTracker`
  Map on every `updateFilePosition`, though `sortedPositions` + `findClosestPositionIndex`
  (binary search) already exist. On multi-million-line files scrubbing the timeline burns CPU.
- **Fix:** use `findClosestPositionIndex` + `positionTracker.get(sortedPositions[idx])` in both
  methods.
- **Verify:** scrubbing a large file stays smooth; behavior identical on small files.

### 29. Context loss permanently kills the viewer
- **File:** [src/viewer.ts:124-126](src/viewer.ts#L124-L126)
- **Problem:** `doNotHandleContextLost: true` means a WebGL context loss (mobile tab switch, GPU
  reset) leaves a dead canvas until full reload.
- **Fix:** drop the flag (accept the small memory overhead) or add an explicit
  `onContextLost/Restored` handler that reloads the current file. Decide and document.
- **Verify:** simulate context loss (`WEBGL_lose_context` extension) → viewer recovers.

### 30. Shader & dead-code hygiene
- **Files:** [src/gpupicker.ts:121](src/gpupicker.ts#L121), [src/gpupicker.ts:162](src/gpupicker.ts#L162), [src/rendershaders.ts:3](src/rendershaders.ts#L3), [src/rendershaders.ts:40](src/rendershaders.ts#L40), [src/lineshader.ts:216-227](src/lineshader.ts#L216-L227), [src/modelmaterial.ts](src/modelmaterial.ts), [src/processor.ts:561-626](src/processor.ts#L561-L626)
- **Problem:** `#if defined(WEBGL2) || defines(WEBGPU)` — `defines` is a typo for `defined`
  (works only because the current preprocessor is lenient); `baseColor` is listed twice in the
  line-shader attribute array; `modelmaterial.ts` and `rendershaders.ts` are entirely unused;
  `testRenderScene` (non-progressive) is dead; heavy `console.log` noise throughout
  (`viewer.ts:430`, `processor.ts` emoji logs, `gpupicker.ts:30`).
- **Fix:** correct the typo, dedupe the attribute, delete dead files/methods, and put logging
  behind a `debug` flag.
- **Verify:** `npm run check` + a load smoke test; grep shows no `modelmaterial` references.

### 31. WASM asset delivery through the inline worker needs a verification pass
- **Files:** [package.json](package.json) (`exports` maps only `./assets/*.js`), [src/viewer-proxy.ts:1](src/viewer-proxy.ts#L1) (`?worker&inline`), [src/wasmprocessor.ts:1](src/wasmprocessor.ts#L1)
- **Problem (needs verification, likely broken in consumers):** the worker is inlined
  (blob/data URL), and wasm-bindgen's `init()` resolves `gcode_file_processor_bg.wasm` relative
  to `import.meta.url` — inside an inlined worker in a *consuming* app (DWC), that base URL is a
  blob URL and the `.wasm` fetch will 404. The package `exports` also only exposes
  `./assets/*.js`, so even a correct URL may be unreachable via the package.
- **Fix:** build the wasm as a bundled asset with an explicit URL
  (`new URL('...wasm', import.meta.url)` handled by vite at library build), or accept an
  explicit wasm URL/ArrayBuffer parameter on `enableWasmProcessing()` that DWC can supply.
  Widen the `exports` map (`"./assets/*": "./dist/assets/*"`). Test in an actual consumer build,
  not just `vue_test` (which aliases the source).
- **Verify:** `npm pack` the library, install it into a scratch vite app, enable WASM, load a
  file — WASM path activates (check `processingStats.method === 'hybrid'`).

### 32. No devicePixelRatio handling (blurry rendering on HiDPI)
- **Files:** [src/viewer-proxy.ts:82-100](src/viewer-proxy.ts#L82-L100), [src/viewer.ts:102-118](src/viewer.ts#L102-L118)
- **Problem:** canvas backing store is sized to CSS pixels; on DPR>1 displays the render is
  upscaled/blurry. If fixed, the GPU picker's pointer→pixel mapping
  ([src/gpupicker.ts:62-63](src/gpupicker.ts#L62-L63)) must be scaled by the same factor.
- **Fix:** multiply width/height by `window.devicePixelRatio` in the proxy resize path (send DPR
  to the worker), and scale `scene.pointerX/Y` in the picker read accordingly. Optional; do
  after P1 items so picking math is touched once.
- **Verify:** sharp rendering on a HiDPI display; hover-picking still hits the right line.

## Implementation notes (post-hoc, from actually doing items 1–28, 30)

- **Item 1** ended up as a `WASM_FileProcessor/pkg-fallback/` (tracked in git) copied into the
  gitignored `pkg/` by `WASM_FileProcessor/ensure-pkg.mjs`, invoked both from `vite.config.mjs`
  (at config-load time, *before* any plugin — `vite-plugin-dts` runs its own TS diagnostics ahead
  of the wasm plugin's `buildStart`, so the fallback has to exist before that too) and from a new
  `precheck` npm script for bare `tsc`. A pure ambient `.d.ts` declaration would have satisfied
  `tsc` but not Rollup's actual module resolution during `vite build`, so it had to be a real,
  importable (if non-functional) stub module. `Processor.enableWasmProcessing()` was also
  tightened so a failed `initialize()` doesn't leave `wasmProcessor` non-null (which would
  otherwise wedge `isWasmEnabled()`/`loadFile()` into a broken "enabled but never works" state
  and make retries impossible).
- **Item 8's mechanism** ended up different from (simpler than) what the plan sketched: instead
  of guarding every `gpuPicker` call site with `?.`, `GPUPicker` never becomes null — it gained
  an `enabled` flag; disabling removes its render target from `scene.customRenderTargets` and
  clears its render list, so the picker goes quiet without ever being torn down or dereferenced
  through a nullable path. That sidesteps the "guard 9 call sites" work entirely. A `setPickingEnabled()`
  API was added end-to-end (Viewer → worker message → both `ViewerProxy` and `ViewerDirect`), plus
  a try/finally around `loadFile` as a second, independent safety net.
- **Item 10's fix** went further than "register the active mesh set" — it also deleted the
  force-enable/restore dance in `GPUPicker`'s render-target observables entirely, since the
  picker's render list now only ever contains meshes that are already enabled (the visible mesh
  mode), so there's nothing left to force-toggle.
- **Item 9**: kept `refreshRate = 1` (read every render) rather than gating the pixel readback on
  pointer movement — gating it would leave the hover highlight stale while the user orbits the
  camera with the pointer still, and I can't browser-test that trade-off. The leak, GL state, and
  crash-safety fixes (single reused framebuffer + `dispose()`, save/restore the framebuffer
  binding, null-guard the private-API texture access, `generateMipMaps: false`, wrap the whole
  read in try/catch) are all still done.
- **Item 18**: also stripped inline comments before tokenizing in `g2g3.ts` (not just the E-token
  check) — a comment word starting with one of the token letters (e.g. `; Edge case`) would
  otherwise corrupt tokenization for the whole line, not just extrusion detection.
- **Item 22**: kept `PrusaSlicer.processHeader()` in place (just fixed the inverted comparison)
  rather than deleting it, since `Cura`/`SuperSlicer`/`OrcaSlicer`/`KiriMoto`/`ideaMaker` all
  define the same no-op override — deleting only PrusaSlicer's would have made it the odd one out
  in that interface. It's still never called by anything; wiring it in is a separate, larger
  change (the streaming line-by-line parser doesn't currently keep the full file around as an
  array to call it with).
- **Item 26**: `ViewerDirect` was brought to full parity and both classes now `implements` a new
  shared `src/viewer-api.ts` interface (so `tsc` catches future drift), but **`ViewerDirect` is
  deliberately not exported from `src/index.ts`**. Exporting it pulls `viewer.ts` (and Babylon's
  own internal dynamic imports) into the main library entry point's reachability graph, and the
  `vite build` output changed from two files (`index.es.js`/`index.cjs.js`) to those two plus
  ~50 extra chunk files — which the package's `exports` map (only `"./assets/*"`) doesn't account
  for and which I have no way to verify actually resolve correctly for a real consumer. Reverted
  that export; `ViewerDirect` is fully implemented and typechecked, just not part of the public
  package surface yet. Whoever wants to expose it should verify the packaging implications (and
  probably widen the `exports` map) in a real consumer build first.
- **Not done (needs a real browser to verify safely):** item 29 (context-loss recovery — behavior
  change with no way to test `WEBGL_lose_context` here), item 31 (WASM asset delivery through the
  inlined worker — needs testing inside an actual DWC-like consumer, not `vue_test`, which aliases
  source directly), item 32 (devicePixelRatio handling — a rendering change I can't visually
  confirm). Also left the "gate all `console.log` noise behind a debug flag" half of item 30 as
  scope: fixed the two call sites the plan named explicitly (`gpupicker.ts`'s constructor log,
  `viewer.ts`'s `setMaxFPS`) plus deleted the two confirmed-dead files/one dead method, but didn't
  do a sweeping pass over every log line in `processor.ts` — most of those are informative
  WASM/perf stats, not noise, and gating them needs a design decision (a `debug` flag on which
  class?) rather than a mechanical fix.
- **Dependencies**: all npm deps were already at latest (`@babylonjs/core`/`materials` 9.15.0,
  `@tweenjs/tween.js` 25.0.0, `typescript` 6.0.3, `vite` 8.1.3, `vite-plugin-dts` 5.0.3) — no
  changes needed. `Cargo.toml`'s deps are open-ended semver ranges with no committed `Cargo.lock`,
  so they already resolve to latest-compatible at build time; the one exception was
  `serde-wasm-bindgen`, which was declared but never referenced anywhere in the Rust source, so
  it was removed rather than version-bumped (bumping 0.4→0.6 is a semver-major jump for a pre-1.0
  crate, and I have no way to compile-check it here).
- **`build.bat`** (repo root) added per request: installs npm deps if missing, attempts the WASM
  build if `wasm-pack` is on PATH (skips gracefully with an explanatory message if not), then runs
  `npm run check` and `npm run build`. Verified end-to-end on this machine (no Rust toolchain) via
  PowerShell — succeeds and produces the same two-file `dist/` output as a plain `npm run build`.
